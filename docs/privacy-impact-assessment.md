# Privacy Impact Assessment для системи моніторингу якості повітря

## 1. Короткий висновок

Система моніторингу якості повітря збирає переважно екологічні дані, однак прив'язка сенсорів до точних координат, технічних ідентифікаторів і журналів обслуговування може створювати privacy-ризики. Основний ризик не пов'язаний із медичними даними, а з можливістю ідентифікувати місце встановлення приватних сенсорів, маршрут обслуговування технічного персоналу або непрямо визначати характеристики домогосподарств чи установ.

## 2. Які дані збираються

Система збирає:

- екологічні показники: `PM2.5`, `PM10`, `CO2`, `NO2`, температура, вологість
- технічні метадані: `sensor_id`, версія firmware, heartbeat, статус батареї
- геодані: координати сенсора або зони розміщення
- операційні журнали: час доставки, помилки зв'язку, історія OTA

## 3. Чи є персональні дані

Так, частково.

- Якщо сенсори стоять лише у публічних місцях, більшість даних не є персональними.
- Якщо сенсори встановлено на приватних будинках, підприємствах або в ОСББ, точна геолокація і технічний ідентифікатор можуть стати персональними або псевдоперсональними даними.
- Дані про обслуговування, якщо вони містять ПІБ технічного працівника, також є персональними.

Отже, точні координати, журнали доступу та сервісні метадані потрібно обробляти як дані з підвищеним privacy-ризиком.

## 4. Правова база

Для України застосовується:

- Закон України `Про захист персональних даних`

Якщо система передає або реплікує дані в інфраструктуру, розміщену в ЄС, або обслуговує європейських користувачів, додатково релевантний:

- `GDPR`

Особливо важливі принципи:

- `lawfulness, fairness, transparency`
- `purpose limitation`
- `data minimization`
- `storage limitation`
- `integrity and confidentiality`

## 5. Основні privacy-ризики

| Ризик | Опис | Рівень |
|---|---|---|
| Ідентифікація приватної локації | точні координати можуть показати адресу приватного сенсора | Medium |
| Надлишкові журнали обслуговування | логи можуть містити зайві ідентифікатори або персонал техпідтримки | Medium |
| Вторинне використання даних | координати і показники можуть використовуватись не лише для екологічного моніторингу | Medium |
| Надмірне зберігання | довге зберігання точних координат збільшує ризик деанонімізації | Medium |
| Несанкціонований доступ | відкриті dashboard або API можуть розкрити точне місце розміщення сенсорів | High |

## 6. Заходи зменшення ризиків

- використовувати зональне відображення замість точних координат у публічному dashboard
- розділити відкриті дані і внутрішні технічні метадані
- не зберігати ПІБ технічного персоналу в телеметричних потоках
- застосовувати рольову модель доступу до координат і OTA-логів
- встановити терміни retention: точні координати і сервісні логи зберігати лише настільки довго, наскільки це потрібно для експлуатації

## 7. Data minimization та purpose limitation

### Data minimization

- для публічного відображення достатньо `zone_id`, а не повної адреси
- у відкритих API не публікувати `sensor_id`, якщо він дозволяє ідентифікувати приватну власність
- журнали інцидентів мають містити технічний код, а не персональні атрибути оператора

### Purpose limitation

- дані використовуються лише для моніторингу якості повітря, аналітики та експлуатації мережі
- забороняється використовувати координати сенсорів для маркетингу, профілювання домогосподарств або непрямого оцінювання поведінки мешканців

## 8. Інформування громадян

Рекомендовані механізми:

- публічна сторінка проєкту з описом, які дані збираються і для чого
- короткі таблички або QR-коди біля сенсорів у публічних місцях
- окремий розділ `Privacy` у міському dashboard
- контакт для подання запиту на видалення або уточнення даних, якщо сенсор пов'язаний із приватною локацією

## 9. Підсумок

Проєкт можна вважати сумісним з принципами `Privacy by Design`, якщо точна геолокація не публікується без потреби, внутрішні технічні журнали захищені, а публічний dashboard працює з агрегованими або зональними даними. У такій конфігурації privacy-ризик знижується до прийнятного рівня.
